# Project Portfolio Guide

Build a portfolio that stands out, showcases your skills, and gets you interviews. This guide covers choosing projects, building them right, and presenting them effectively.

---

## Why Your Portfolio Matters

**Reality Check:**
- Recruiters spend 6 seconds on your portfolio initially
- Live demos are 10x more impressive than descriptions
- 1 exceptional project beats 5 mediocre ones
- Your portfolio is your proof of work when you lack professional experience

**What Employers Look For:**
- ✅ Can you build complete, working applications?
- ✅ Do you write clean, maintainable code?
- ✅ Can you deploy and manage production apps?
- ✅ Do you understand UX and design basics?
- ✅ Can you explain your technical decisions?

---

## Section 1: Choosing the Right Projects

### What Impresses Employers

**Projects That Stand Out:**

1. **Solves a Real Problem**
   - ✅ Budget tracker that YOU actually use
   - ✅ Tool that automates something boring
   - ❌ Generic todo list that you'll never use

2. **Shows Technical Depth**
   - ✅ Real-time features (WebSockets, optimistic updates)
   - ✅ Complex state management
   - ✅ API integration with error handling
   - ❌ Static site with no interactivity

3. **Demonstrates Production Skills**
   - ✅ Deployed with CI/CD
   - ✅ Error monitoring (Sentry)
   - ✅ Analytics integration
   - ✅ Loading states, error boundaries
   - ❌ Works on localhost only

4. **Shows Design Sensibility**
   - ✅ Responsive, accessible, polished UI
   - ✅ Consistent design system
   - ❌ Unstyled or poorly designed

**Red Flags to Avoid:**
- ❌ Tutorial projects (everyone has them)
- ❌ Broken deployments
- ❌ No README or documentation
- ❌ Dead links or 404 pages
- ❌ "Coming soon" projects
- ❌ Unfinished projects without explanation

**Code Snippet: Custom Cart Hook**
```jsx
// hooks/useCart.js
import { useState, useEffect } from 'react';

export function useCart() {
  const [cart, setCart] = useState(() => {
    const saved = localStorage.getItem('cart');
    return saved ? JSON.parse(saved) : [];
  });

  useEffect(() => {
    localStorage.setItem('cart', JSON.stringify(cart));
  }, [cart]);

  const addToCart = (product, quantity = 1) => {
    setCart(prev => {
      const existing = prev.find(item => item.id === product.id);
      if (existing) {
        return prev.map(item =>
          item.id === product.id
            ? { ...item, quantity: item.quantity + quantity }
            : item
        );
      }
      return [...prev, { ...product, quantity }];
    });
  };

  const removeFromCart = (productId) => {
    setCart(prev => prev.filter(item => item.id !== productId));
  };

  const updateQuantity = (productId, quantity) => {
    setCart(prev =>
      prev.map(item =>
        item.id === productId ? { ...item, quantity } : item
      )
    );
  };

  const clearCart = () => setCart([]);

  const total = cart.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const itemCount = cart.reduce((sum, item) => sum + item.quantity, 0);

  return {
    cart,
    addToCart,
    removeFromCart,
    updateQuantity,
    clearCart,
    total,
    itemCount
  };
}
```

```jsx
    setLoading(false);
  }
};
```

**❌ Don't Include:**
- Broken features or incomplete sections
- Lorem ipsum placeholder text
- Default "Create React App" favicon
- Console errors or warnings
- Unhandled errors that crash the app
- Massive un-optimized images
- Hard-coded API keys (use environment variables!)
- "Under construction" pages
- Links that go nowhere

**✅ Instead:**
- Only deploy features that work
- Write real content (even if brief)
- Custom favicon and meta tags
- Clean console (no warnings in production)
- Error boundaries to catch crashes
- Optimized, lazy-loaded images
- Environment variables for sensitive data
- Remove unfinished features or mark them clearly
- Ensure all navigation works

## 🎯 Key Technical Decisions

Explain interesting technical choices:

**Why WebSockets instead of polling?**
With 100+ concurrent users, polling every 5 seconds would generate
12,000 requests/minute. WebSockets reduced server load by 95% while
providing instant updates.

**Why Zustand over Redux?**
Redux felt like overkill for this app's state complexity. Zustand
provided the same DX with 80% less boilerplate.

## Section 5: Deployment Guide

### Platform Comparison

| Platform | Best For | Free Tier | Pros | Cons |
|----------|----------|-----------|------|------|
| **Vercel** | Next.js, React SPAs | Generous | Zero-config, fast, great DX | Limited backend support |
| **Netlify** | React SPAs, static sites | Generous | Simple, form handling, serverless functions | Slower builds than Vercel |
| **GitHub Pages** | Static sites, portfolios | Free (public repos) | Simple, integrated with GitHub | Static only, no server-side |
| **Railway** | Full-stack, backend APIs | $5/month included | Easy database setup, Docker support | Not free |
| **Render** | Full-stack apps | Free (with limitations) | Supports Node, Python, Docker | Slow cold starts on free tier |

### Vercel Deployment (Step-by-Step)

**For Create React App / Vite:**

1. **Push your code to GitHub**
   ```bash
   git init
   git add .
   git commit -m "Initial commit"
   git branch -M main
   git remote add origin https://github.com/yourusername/your-repo.git
   git push -u origin main
   ```

2. **Go to [Vercel](https://vercel.com) and sign up with GitHub**

3. **Click "New Project"**
   - Select your repository
   - Vercel auto-detects framework (React, Next.js, etc.)

4. **Configure Build Settings** (usually auto-detected)
   - Framework Preset: Create React App (or Vite)
   - Build Command: `npm run build`
   - Output Directory: `build` (or `dist` for Vite)
   - Install Command: `npm install`

5. **Add Environment Variables**
   - Click "Environment Variables"
   - Add each variable from your `.env` file
   - Example: `REACT_APP_API_URL`, `REACT_APP_FIREBASE_KEY`

6. **Click "Deploy"**
   - Wait 1-2 minutes
   - Your app is live at `your-project.vercel.app`

7. **Set Up Custom Domain** (optional)
   - Go to Settings → Domains
   - Add your domain
   - Update DNS records as instructed

**Auto-Deploy on Push:**
- Every push to `main` branch auto-deploys
- Pull requests get preview deployments
- Rollback to previous deployment in one click

---

### Netlify Deployment (Step-by-Step)

**Option 1: Drag and Drop**

1. Build your app locally:
   ```bash
   npm run build
   ```

2. Go to [Netlify](https://netlify.com) → Sites

3. Drag your `build` folder to the drop zone

4. Your site is live at `random-name-123.netlify.app`

**Option 2: Git Integration (Recommended)**

1. Push code to GitHub

2. Go to Netlify → "Add new site" → "Import an existing project"

3. Connect to GitHub and select your repo

4. Configure build settings:
   - Build command: `npm run build`
   - Publish directory: `build`

5. Add environment variables in "Site settings" → "Environment variables"

6. Click "Deploy site"

**Netlify Features:**
- Form handling (no backend needed!)
- Serverless functions (`/netlify/functions`)
- Split testing (A/B testing)
- Analytics

---

### GitHub Pages Deployment (Step-by-Step)

**For Static Sites Only** (no server-side logic)

1. Install gh-pages package:
   ```bash
   npm install --save-dev gh-pages
   ```

2. Add to `package.json`:
   ```json
   {
     "homepage": "https://yourusername.github.io/your-repo",
     "scripts": {
       "predeploy": "npm run build",
       "deploy": "gh-pages -d build"
     }
   }
   ```

3. Deploy:
   ```bash
   npm run deploy
   ```

4. Enable GitHub Pages:
   - Go to repo Settings → Pages
   - Source: Deploy from a branch
   - Branch: `gh-pages`
   - Click Save

5. Site will be live at `yourusername.github.io/your-repo`

**Limitations:**
- Static only (no API routes)
- Client-side routing needs special config
- For CRA, add to `public/` a `404.html` that redirects to `index.html`

### Environment Variables Best Practices

**❌ Don't:**
```javascript
// Hardcoded API key in code
const API_KEY = "sk_live_abc123xyz";

fetch(`https://api.example.com/data?key=${API_KEY}`)
```

**✅ Do:**
```javascript
// Use environment variables
const API_KEY = import.meta.env.VITE_API_KEY; // Vite
// or
const API_KEY = process.env.REACT_APP_API_KEY; // Create React App

fetch(`https://api.example.com/data?key=${API_KEY}`)
```

**File Structure:**
```
# .env.local (don't commit!)
VITE_API_URL=http://localhost:3000
VITE_STRIPE_KEY=pk_test_abc123

# .env.example (commit this!)
VITE_API_URL=your_api_url_here
VITE_STRIPE_KEY=your_stripe_key_here

# .gitignore
.env
.env.local
.env.production.local
```

**On Vercel/Netlify:**
- Add variables in dashboard
- Mark sensitive ones as "Secret" (encrypted)
- Variables are injected at build time

```jsx
// components/FlashcardDeck.jsx
import { useFlashcards } from '@/hooks/useFlashcards';
import { calculateNextReview } from '@/lib/spacedRepetition';
import { Card } from '@/components/ui/Card';

export function FlashcardDeck({ deckId }) {
  const { cards, loading, error, updateCard } = useFlashcards(deckId);
  const [currentIndex, setCurrentIndex] = useState(0);

  const handleAnswer = async (difficulty) => {
    const card = cards[currentIndex];
    const nextReview = calculateNextReview(card, difficulty);

    await updateCard(card.id, { nextReview, lastReviewed: Date.now() });
    setCurrentIndex(i => i + 1);
  };

  if (loading) return <Skeleton />;
  if (error) return <ErrorMessage error={error} />;
  if (currentIndex >= cards.length) return <CompletionScreen />;

  return (
    <Card>
      <Flashcard card={cards[currentIndex]} />
      <DifficultyButtons onAnswer={handleAnswer} />
    </Card>
  );
}
```
